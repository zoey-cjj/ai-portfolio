📘 Case 1：族谱智能问答助手（RAG）

1️⃣ 项目背景 & 问题定义

族谱多为 PDF / 扫描件 / 纯文本，结构复杂、同名人多、古文门槛高。用户无法用关键词搜索回答“我是第几代？”这类亲缘问题。

  • 产品目标：自然语言提问，降低使用门槛
  • 成功指标：自动解决率 ≥ 70%，幻觉率 ≤ 5%，平均响应 < 3s

2️⃣ 为什么必须用 AI？

规则引擎无法覆盖复杂亲缘语义；人工解答不可扩展。RAG 可在不微调的前提下引入私有知识，并返回原文片段作为依据。

3️⃣ 方案设计

用户输入 → 意图识别 → 检索（RAG）→ LLM 生成 → 安全过滤 → 输出

4️⃣ Prompt 设计思路

System Prompt 明确角色为“族谱专家”，要求仅基于上下文作答，无法回答时声明“知识库未收录”。

5️⃣ 评测体系（Eval）

准确性 / 完整性 / 幻觉 / 合规性，基于 20 条 Golden Set 验证。

📘 Case 2：中东支付风控助手（AI + 规则融合）

1️⃣ 业务背景

中东地区支付监管复杂，拒付与欺诈风险高，人工审核成本高、误杀率偏高。

2️⃣ 产品目标

  • 降低人工审核成本
  • 提升高风险交易识别准确率
  • 减少正常交易误杀率

3️⃣ 为什么引入 AI？

规则只能覆盖已知风险模式；LLM 可用于解读复杂风控规则与交易上下文。

4️⃣ 方案设计

交易接入 → 规则初筛 → LLM 风险分析 → 人工审核 → 决策

5️⃣ AI 产品边界

  • AI 不直接拦截交易
  • AI 仅作为“建议方”
  • 所有建议需可审计

🧪 评测体系（Eval）

用于衡量族谱问答系统的回答质量，指导上线与迭代。

分数 标准
5 完全正确，无幻觉，来源清晰
3 部分正确，或缺少关键依据
1 错误 / 有害 / 明显幻觉

目标：幻觉率 ≤ 5%，自动解决率 ≥ 70%（基于 20 条 Golden Set）
